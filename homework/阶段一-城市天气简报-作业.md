# 阶段一作业：城市天气简报 CLI

> LangChain.js 学习用个人作业，完成后可放进简历项目经历作补充说明。

## 目标

在**不依赖本文件逐步教程**的前提下，自行查阅 [LangChain 文档](https://docs.langchain.com/)（选择 **JavaScript / TypeScript**），实现一个可运行的小程序，覆盖：**多轮对话、真实 HTTP Tool、结构化输出、基础错误与超时**。

## 任务说明

### 功能要求

1. **多轮对话**  
   使用 `readline`（或等价方式）循环读取用户输入；用户输入 `quit` 或 `exit` 退出程序。

2. **至少 1 个真实 HTTP Tool**  
   调用**无需 API Key** 的公开天气相关接口。推荐组合：  
   - [Open-Meteo Geocoding API](https://open-meteo.com/en/docs/geocoding-api)（城市名 → 经纬度）  
   - [Open-Meteo Forecast API](https://open-meteo.com/en/docs)（经纬度 → 预报）  

   用户可以用自然语言提问（例如「上海明天适合出门吗」），由模型决定何时调用 tool；tool 内完成 HTTP 请求并返回可被模型使用的文本或 JSON 字符串。

3. **结构化输出（强制）**  
   在面向用户的最终答复路径上，必须有一段 **Schema 约束的结构化结果**，例如：

   | 字段 | 说明 |
   |------|------|
   | `city` | 解析到的城市名（或展示用名称） |
   | `latitude` | 纬度 |
   | `longitude` | 经度 |
   | `summary` | 天气要点摘要（基于 tool 事实，勿编造） |
   | `recommendation` | 枚举：`适合` \| `不太适合` \| `不确定` |

   实现方式任选：如 `withStructuredOutput`、官方文档中的 structured output 模式等（可与现有 `structured-output.ts` 思路对齐）。

4. **错误处理**  
   - HTTP 请求使用超时（建议约 **8s**，`AbortController`）。  
   - 城市无法解析、接口 4xx/5xx 或网络错误时：**进程不崩溃**，由助手用自然语言说明原因。

### 技术要求

- 使用本仓库已有的 **`createModel`**（`src/lib/model.ts`）或等价封装，通过 `.env` 配置 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL` 等（与 `chat-tool-cli` 一致即可）。
- 在 `package.json` 中增加脚本，例如：`phase1:weather`，便于一键运行。

### 自测清单（提交前自行跑通）

- [ ] 输入「北京」或「上海」等，能产出含结构化字段的简报。  
- [ ] 输入明显胡编的地名，有友好错误提示且不崩溃。  
- [ ] 连续两轮不同城市，对话仍合理（可对历史做条数截断，参考 `chat-tool-cli` 的 `trimHistory` 思路）。

### 验收标准（对照即可）

- [ ] 多轮 CLI + 退出命令  
- [ ] 真实 HTTP tool（Open-Meteo 或等价）+ 超时  
- [ ] 至少一处 **Schema 约束的结构化输出**（TypeScript 类型或 Zod 等）  
- [ ] 失败路径不崩溃  

## 建议实现顺序（可自行调整）

1. 先打通 Open-Meteo 的 `fetch`（在独立函数或 tool 里），确认无 Key 可访问。  
2. 接上 `ChatOpenAI` + `bindTools`，能触发 tool 调用。  
3. 再接入结构化输出（注意与 tool 调用的先后顺序，文档中任选一种清晰方案并在代码注释里写一句「为何这样设计」）。  
4. 最后补全多轮历史与裁剪、错误文案。
